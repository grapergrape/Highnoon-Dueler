# Combat Presentation UX Plan

Status: planned. This document describes the approved UX direction only. Do not implement until the next explicit implementation confirmation.

The goal is to move combat from a functional card list into a physical western card duel: a fanned hand, readable card previews, tactile play animations, and immediate combat feedback. This should be done with browser-native JavaScript, DOM, CSS, and the existing canvas effects layer. A full game engine is not needed for this phase.

## Decision

Use a custom presentation layer, not a standalone game engine.

Reasons:

- The game is turn/state/card driven, not real-time physics driven.
- DOM and CSS are stronger for card text, layout, hover states, buttons, and fast iteration.
- The existing duel rules are already isolated in `static/js/duel/duel.js`.
- The existing canvas renderer can keep handling scene art, Showdown overlays, tracers, dust, flashes, and screen shake.
- A presentation layer gives most of the perceived quality gain without rewriting the app.

## Target Feel

Combat should feel like sitting across from an outlaw with a rough poker hand.

Player-facing goals:

- The hand looks like cards held in the player's hand, not a wrapped grid.
- Cards are rough western poker cards: worn edges, stained parchment, ink stamps, brass cost coin, suit-like marks by type, and distressed texture.
- Hovering a card makes the decision clearer before committing.
- Playing a card is a short physical action: card leaves the hand, strikes the table, resolves, then disappears.
- The table always shows what matters now: loaded bullets, Position, Armor, Nerve, incoming HP loss, outgoing damage, and whether the current round is clean.
- The perfect-run chase is visible without becoming tutorial copy.

This is a desktop-first Steam target. Mobile can remain serviceable, but the first quality pass should optimize for `1366x768`, `1440x900`, and `1920x1080`.

## Proposed Folder

Add:

```text
static/js/presentation/
  combat-presenter.js
  hand-layout.js
  card-preview.js
  card-animations.js
  presentation-events.js
```

Ownership:

- `combat-presenter.js`
  - Owns combat UI timing.
  - Gates input during card-play and Showdown transitions.
  - Coordinates ghost-card animation, mechanics callback, UI refresh, and feedback pulses.
  - Does not own combat rules.

- `hand-layout.js`
  - Computes fanned hand transforms.
  - Handles compression for large hands.
  - Exposes stable CSS variables for card position, rotation, scale, and z-index.

- `card-preview.js`
  - Computes hover preview deltas for one candidate card.
  - Must not mutate duel or run state.
  - Should call a pure duel preview helper rather than duplicating card logic.

- `card-animations.js`
  - Owns DOM ghost-card creation and animation.
  - Handles play, reject, burn/fade, impact pulse, and hand reflow animation.

- `presentation-events.js`
  - Defines small event names used by the presenter and future sound layer.
  - Keeps animation/sound hooks consistent without creating a full engine.

## Dependency Rules

Keep mechanics authoritative in `duel/`.

Allowed dependency direction:

```text
app -> presentation -> duel preview helpers
app -> ui
app -> rendering
ui -> duel display helpers
rendering -> duel display helpers
```

Important rules:

- `presentation/` may read DOM generated by `ui.js`.
- `presentation/` may call callbacks provided by `app.js`.
- `presentation/` must not call `tryPlayCard` directly unless `app.js` passes that operation as a callback.
- `presentation/` must not save runs, advance screens, or modify persistent run state.
- `presentation/` must not become a second rule engine.

If hover preview needs exact card resolution, add a pure helper to `duel/duel.js`, for example:

```js
previewCardPlay(duel, run, uid)
```

That helper should return before/after values and reasons without mutating either object.

## Hand System

Replace the combat hand's flat wrapping row with a dedicated table/hand structure.

Target markup shape:

```html
<div class="combat-table">
  <div class="play-focus-zone" id="play-focus-zone"></div>
  <div class="hand-rail" id="hand"></div>
</div>
```

Each combat card should be a real button:

```html
<button
  class="hand-card hand-card-feat"
  data-card-uid="..."
  data-card-id="..."
  data-card-type="feat"
  data-affordable="true"
  style="--hand-x: 0px; --hand-y: 0px; --hand-r: 0deg; --hand-z: 1;"
>
  ...
</button>
```

The current combat cards are clickable `div`s. The implementation pass should move them to buttons for better input semantics, focus states, disabled states, and keyboard/controller readiness.

### Layout Rules

For `n` cards:

- Center card sits lowest rotation.
- Outer cards rotate more and sit slightly lower.
- Hovered card rises, rotates toward zero, scales up, and takes top z-index.
- Large hands compress spacing before shrinking text.
- The hand rail should reserve enough height so hover lift does not shift the whole panel.

Starter formula:

```js
const center = (count - 1) / 2;
const offset = index - center;
const spread = Math.max(72, 118 - count * 5);
const rotation = offset * 4.5;
const x = offset * spread;
const y = Math.abs(offset) * 5;
```

Tune separately for:

- `1-3` cards: loose, readable.
- `4-7` cards: Hearthstone-like fan.
- `8-10` cards: compressed but still hover-readable.
- `11+` cards: emergency compression, stronger hover zoom.

## Card Visual Design

Use the existing card markup as the base, but make cards feel like western poker objects.

Card anatomy:

```text
corner cost coin
type stamp / rarity strip
card name
western symbol or illustration well
flavor line
rule/effect chips
build-path stamp
```

Style direction:

- parchment base, not clean white
- dark ink name text
- irregular black/brown outer edge
- worn corner clipping
- faint stains and scratches
- subtle paper grain overlay
- brass cost coin
- stamped type ribbon
- class/build-path badge as a ledger stamp

Type language:

| Type | Visual read |
| --- | --- |
| Gun | iron, rust, dark leather, heavier border |
| Feat | blood red, brass, dirty ink |
| Stance | green leather, steadier border |
| Showdown/legacy | faded blue steel, inactive/rare treatment |
| Item/trinket/shop | tan ledger, merchant stamp |

Rarity should affect trim and ornamentation, not turn cards into bright fantasy panels. Legendary can use brass filigree and stronger distress, but the western table should stay grounded.

CSS approach:

- Use `::before` for paper grain, scratches, and stains.
- Use `::after` for worn edge darkening.
- Use CSS variables for accent, rarity trim, hover glow, and disabled wash.
- Keep cards DOM-based so text remains crisp and easy to tune.

## Card Play Animation

Played cards do not stay as minions. They are tricks thrown onto the table.

Flow:

1. Player clicks an affordable card.
2. Presenter locks combat input.
3. Presenter captures the clicked card's DOM rectangle.
4. Presenter clones it into a fixed `.presentation-layer` as a ghost card.
5. Original hand card fades or collapses.
6. Ghost card flies to the table focus zone.
7. On impact, `app.js` executes the real card play callback.
8. UI refreshes.
9. Combat floats/stat pulses trigger.
10. Ghost card burns, slides, or dust-fades away.
11. Presenter reflows the new hand and unlocks input.

Recommended timing:

| Moment | Duration |
| --- | ---: |
| click squash | 60 ms |
| fly to table | 180-260 ms |
| impact hold | 80-120 ms |
| burn/fade out | 160-220 ms |
| hand reflow | 120-180 ms |

Different card families can vary slightly:

- Gun: heavier snap, metallic pulse, chamber/cylinder accent.
- Armor/defense: lower impact, shield/ledger stamp pulse.
- Load/damage: sharper throw, muzzle flash/bullet click.
- Free card: gold-green flash at the cost coin.
- Rejected card: short shake plus red cost/HP pulse.

## Hover Preview

Hover preview is the highest-value UX upgrade because it improves decision quality.

When hovering a card, show deltas directly in the combat dashboard:

```text
Loaded: 2/6 -> 4/6
Position: 1 -> 3
Armor: 0 -> 5
Outgoing: 12 -> 21
Incoming: 8 -> 5
Nerve: 3 -> 1
```

Rules:

- Preview must be exact whenever possible.
- Preview must not mutate state.
- Unaffordable cards should show why: not enough Nerve, lethal HP payment, disabled by card rule, or no valid target.
- The values affected by the hovered card should pulse or outline.
- Cards with mixed consequences must show both upside and downside.

Examples:

- `load+N`: pulse loaded bullets and outgoing damage.
- `armor+N`: pulse Armor and incoming HP loss.
- `position+N`: pulse Position, outgoing damage, and any enemy response affected by Position.
- `payHp+N`: pulse HP in red before click.
- `enemyWeak+N`: pulse enemy intent and incoming HP loss.
- `draw+N`: show draw count and Nerve impact, but avoid pretending the drawn cards are known.

## Combat Readability Layout

The combat screen should be organized by decision flow:

```text
enemy scene / canvas
incoming and outgoing forecast overlay
compact round state row
play focus zone
fanned card hand
ledger/log collapsed by default
```

The most important values must remain visible while the hand is hovered:

- current Nerve and max Nerve
- loaded bullets and capacity
- Position and multiplier
- Armor
- outgoing expected damage
- incoming HP loss
- enemy intent
- clean/perfect status

The battle log remains secondary. It should explain what happened after the fact, not be required to decide the turn.

## Perfect-Run Chase Feedback

Add a compact "clean round" strip or meter. It should reward clean planning without using tutorial paragraphs.

Possible states:

- `Clean`: no wasted cards and no unblocked HP loss projected.
- `Calm`: within the first three-card calm line.
- `Loaded`: lethal or near-lethal line is forming.
- `Untouched`: incoming HP loss is zero.
- `Cracked`: player crossed the calm line, took projected damage, or paid dangerous HP.
- `Kill Line`: hovered or current state has lethal outgoing damage.

Presentation:

- small brass ticks or stamped words
- crack animation when broken
- stronger pulse when lethal appears
- no big modal, no long instructions

This should make players chase a perfect round/run through feedback, not through explanation.

## CSS/DOM Requirements

Add a top-level presentation layer in `static/index.html` or dynamically from JS:

```html
<div id="presentation-layer" class="presentation-layer" aria-hidden="true"></div>
```

Requirements:

- It must sit above the panel and canvas when animating ghost cards.
- It must not intercept input except during locked presentation moments.
- It must clear itself after each animation.
- Animations should respect `prefers-reduced-motion`.
- Card text must remain within its card at desktop target sizes.
- No UI element should resize the whole combat panel during hover.

## Implementation Order

1. Add presentation folder and the combat presenter shell.
2. Change combat hand cards from `div` to `button` and add stable `data-*` attributes.
3. Add hand rail/fan layout with no mechanics changes.
4. Add western card visual polish and texture layers.
5. Add ghost-card play animation while preserving current immediate card resolution semantics.
6. Move resolution to the impact frame once the animation path is stable.
7. Add exact hover preview with a pure duel preview helper.
8. Add stat delta highlights and rejected-card reasons.
9. Add clean-round/perfect-line feedback.
10. Tune desktop layout and timing with browser smoke tests.

## Acceptance Criteria

The implementation is ready when:

- The hand clearly looks fanned and physical on desktop.
- Hovering any playable card gives useful projected deltas.
- Clicking a card produces a clear play animation before the card disappears.
- The hand reflows without layout jumps.
- The player can understand incoming damage and outgoing damage without opening the log.
- Rejected cards explain why through immediate visual feedback.
- Showdown still feels fast and does not fight the new card timing.
- The perfect-run feedback makes clean lines feel worth chasing.
- Existing duel mechanics and balance sims are not changed except for any pure preview helper needed by the UI.

## Out Of Scope For This Pass

- Rewriting the game in Unity, Godot, Pixi, or a custom engine.
- Full controller navigation.
- Full sound design implementation.
- Artist-final card illustrations.
- Rebalancing card effects.
- Replacing the existing canvas duel scene system.
